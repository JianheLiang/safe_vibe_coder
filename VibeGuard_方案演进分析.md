# VibeGuard 方案演进分析

## 一、原始方案概述

### 项目定位

VibeGuard 最初定位为一个 **CLI 安全拦截层**，部署在 AI 编程助手与本地执行环境之间，对 AI 生成的命令、脚本和 SQL 查询进行实时拦截和风险评估。

### 核心架构设想

- CLI 包装器捕获 AI 生成的命令
- 规则引擎（正则匹配）检测危险模式：秘密泄露、破坏性 SQL、危险 Shell 命令、未授权网络请求
- 检测到风险时阻断执行，要求用户确认
- 可选 LLM 集成，为被拦截的命令建议更安全的替代方案
- Git pre-commit hook 扫描暂存文件中的密钥

### 技术栈

Python 3 + subprocess/os 模块 + 正则表达式 + Claude/OpenAI API（可选）+ Git hooks

## 二、原始方案的可行性质疑

### 2.1 结构性矛盾：拦截点不存在

VibeGuard 的核心假设是可以在 AI 编程工具和执行环境之间插入一个安全层。但实际上：

- **Claude Code、Cursor、Copilot 等主流 AI 编程工具是封闭的端到端架构**，AI 生成的代码在工具内部直接执行，外部程序无法介入执行管道。
- 要实现透明拦截，需要进程注入、系统调用劫持或修改工具本身——这远超高中项目的能力范围，也远超一般开发者的权限。
- **这不是技术难度问题，而是架构上根本不允许。** VibeGuard 想站的那个位置——AI 工具和执行环境之间——在现有架构下不存在一个可以站的缝隙。

### 2.2 退而求其次的路径均不成立

| 替代方案             | 问题                                                                 |
| -------------------- | -------------------------------------------------------------------- |
| 独立审查工具         | 用户需要手动复制粘贴代码进行检查，破坏了 vibe coding 的流畅性        |
| 手动包装 CLI         | `vibeguard run "命令"` 模式增加操作步骤，不方便的工具最终不会被使用  |
| Git pre-commit hook  | 只管提交阶段，管不到实时执行；且该赛道已有 truffleHog 等成熟方案     |

### 2.3 厂商已内置安全机制

主流 AI 编程工具本身已经具备风险审查能力：

- 对高风险 bash 命令弹出确认对话框
- 敏感操作需要用户显式授权
- 部分工具有沙箱执行环境

VibeGuard 试图从外部做的事情，厂商从内部已经做了，而且厂商天然拥有完整的执行上下文，做得更好。

### 2.4 核心结论

> **VibeGuard 的问题不是做不做得出来，而是做出来没有位置放。** 原方案的可行性很低。

## 三、方案转型：从外部拦截到内部嵌入

### 3.1 转型思路

放弃"从外部拦截封闭管道"的路线，转向 **利用 AI 编程工具自身的扩展机制（Skills / Rules / Commands）从内部嵌入安全检查逻辑**。

- Claude Code 支持 `.claude/commands/` 自定义命令
- Cursor 支持 rules 配置
- Copilot 支持 custom instructions

Skills 作为模型决策过程的一部分运行，而非从外面试图拦截。这从根本上解决了原方案的结构性矛盾。

### 3.2 两大核心场景

**场景一：高风险命令筛查**

- 用户自定义高风险命令列表（如 `rm -rf`、`TRUNCATE TABLE`、`DROP DATABASE`、`chmod 777`）
- 在 `--dangerously-skip-permissions` 等模式下，工具自带的权限确认被跳过，此时 skill 层面的审查成为最后一道防线
- 模型在生成命令时拥有完整的意图上下文，可以判断是否有更安全的替代方式

**场景二：上行内容脱敏**

- vibe coding 过程中，项目文件里的数据库密码、`.env` 中的 API Key、配置文件里的个人信息（身份证号、姓名、密码等）可能被模型读取后发送到云端 API
- Skill 在构建上下文阶段对敏感信息进行替换或告警
- 这个需求真实存在且目前缺乏解决方案

## 四、渐进式规则积累机制

### 4.1 冷启动问题

脱敏 skill 面临一个现实问题：不可能一开始就知道用户项目里有哪些敏感信息。

### 4.2 解决方案：渐进式学习

```
第一次交互：
  用户项目中 DB_PASSWORD=xxx123 被发送到 LLM
  → 模型识别到这是敏感字段
  → 自动生成规则：DB_PASSWORD 的值替换为占位符
  → 规则写入本地规则库

后续交互：
  Skill 在构建上下文阶段读取规则库
  → DB_PASSWORD=xxx123 被替换为 DB_PASSWORD=<REDACTED>
  → 替换后的内容发送给 LLM，敏感值不再上传
```

虽然一开始会有敏感信息上行，但随着用户使用模式的固化和识别积累，最终大部分敏感信息都会在本地被脱敏后再发送。**用得越久，规则库越完善，泄露面越小。**

### 4.3 高风险命令规则

高风险命令由用户自行定义，因为"什么是危险命令"在不同项目中差异很大：

- 对数据库管理员来说 `TRUNCATE TABLE` 可能是日常操作
- 对前端开发者来说就是灾难
- 用户自己往规则库里添加条目，比硬编码通用规则更实际

## 五、新方案技术架构

### 5.1 三大组件

```
┌─────────────────────────────────────────────────┐
│              AI 编程工具（Claude Code 等）         │
│                                                   │
│   ┌───────────────┐    ┌──────────────────────┐  │
│   │  VibeGuard    │    │   本地规则库          │  │
│   │  Skills 套件  │◄───│  （JSON / YAML）      │  │
│   │               │    │                      │  │
│   │  - 脱敏 skill │    │  - 敏感字段模式      │  │
│   │  - 命令审查   │    │  - 高风险命令列表    │  │
│   │    skill      │    │  - 用户自定义规则    │  │
│   └───────┬───────┘    └──────────┬───────────┘  │
│           │                       │               │
│           ▼                       ▼               │
│   ┌───────────────────────────────────────────┐  │
│   │         规则学习机制（半自动）              │  │
│   │  模型发现新敏感模式 → 建议 → 用户确认     │  │
│   │  → 写入规则库                              │  │
│   └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 5.2 组件详解

**组件一：本地规则库**

- 格式：JSON 或 YAML 文件
- 内容：敏感字段匹配模式 + 高风险命令黑名单
- 存储：完全在用户本地，不上传到任何云端
- 示例结构：

```yaml
sensitive_fields:
  - pattern: "DB_PASSWORD=.+"
    replacement: "DB_PASSWORD=<REDACTED>"
  - pattern: "api_key:\\s*['\"]sk-[a-zA-Z0-9]+"
    replacement: "api_key: '<REDACTED>'"
  - pattern: "\\b\\d{18}\\b"  # 身份证号
    replacement: "<ID_REDACTED>"

dangerous_commands:
  - pattern: "rm\\s+-rf"
    severity: critical
    message: "检测到递归强制删除命令，请确认目标路径"
  - pattern: "TRUNCATE\\s+TABLE"
    severity: high
    message: "检测到清空表操作，数据将不可恢复"
  - pattern: "DROP\\s+DATABASE"
    severity: critical
    message: "检测到删除数据库操作"
  - pattern: "chmod\\s+777"
    severity: medium
    message: "检测到开放所有权限，存在安全风险"
```

**组件二：Skills 套件**

- 脱敏 Skill：执行时读取规则库，在上下文构建阶段进行文本替换
- 命令审查 Skill：对即将生成的命令与规则库中的高风险模式进行匹配，触发告警或要求确认
- 以 Claude Code 的 `.claude/commands/` 机制为主要实现载体

**组件三：规则学习机制**

- 模型在交互过程中发现新的敏感模式时，提示用户是否加入规则库
- 半自动模式：模型建议 → 用户确认 → 写入规则库
- 避免全自动以防规则膨胀或误判

### 5.3 可选技术路线

| 路线                       | 实现方式                                                       | 难度 | 适用场景                  |
| -------------------------- | -------------------------------------------------------------- | ---- | ------------------------- |
| 路线 A：纯 Skill 指令      | 通过 prompt 指令要求模型遵守规则库中的脱敏和审查规则            | 低   | 快速原型，五周内可完成    |
| 路线 B：Skill + 预处理脚本 | Skill 调用本地 Python 脚本在上下文构建前对文件内容进行正则替换  | 中   | 更可靠的脱敏效果          |
| 路线 C：MCP Server 集成    | 将规则库封装为 MCP Server，模型通过工具调用进行规则查询和更新    | 中高 | 最灵活，支持动态规则管理  |

**建议课程项目选择路线 A 为主，路线 B 为辅。** 路线 A 可以快速验证核心概念，路线 B 解决"skill 只是指令而非强制约束"的可靠性问题。路线 C 更适合作为未来发展方向在报告中讨论。

## 六、方案叙事逻辑的转变

| 维度     | 原方案                                   | 新方案                                   |
| -------- | ---------------------------------------- | ---------------------------------------- |
| 关系     | 对抗关系：从外部管住 AI                   | 协作关系：让 AI 自己学会保护用户          |
| 架构     | 外部拦截封闭管道                          | 内部嵌入工具扩展机制                      |
| 规则来源 | 开发者硬编码通用规则                      | 用户自定义 + 模型渐进式学习积累           |
| 脱敏时机 | 事后检查                                  | 上下文构建阶段预处理                      |
| 用户体验 | 需要改变工作流（手动包装命令）            | 无感嵌入，不改变任何使用习惯              |
| 可行性   | 很低（拦截点不存在）                      | 高（利用现有扩展机制）                    |

## 七、需要注意的局限性

1. **Skill 的执行力边界**：Skill 本质上是对模型的"指令"而不是"强制约束"，模型有可能不遵守。团队应通过测试衡量遵守率，并在报告中坦诚讨论这一局限。

2. **冷启动阶段的泄露**：规则库为空时，第一次出现的敏感信息仍然会被发送到 LLM。这是渐进式方案的固有代价，可以通过预置常见模式（如 API Key 格式、身份证号正则）来缓解。

3. **规则泛化能力**：基于正则的规则可能无法覆盖所有变体。例如密码可能以变量赋值、配置文件、环境变量等多种形式出现。

4. **跨工具兼容性**：不同 AI 编程工具的扩展机制不同，skill 可能需要针对每个工具适配。建议课程项目先聚焦 Claude Code 一个平台。

## 八、课程项目交付建议

### 五周内建议聚焦两件事

1. **脱敏规则的积累和执行流程跑通**：实现规则库 + 脱敏 skill + 手动/半自动规则添加
2. **对比 Demo**：展示有 skill 和无 skill 时的行为差异

### 演示场景建议

- 场景 1：项目中包含 `.env` 文件，其中有数据库密码和 API Key。无 skill 时信息直接发送给 LLM；有 skill 后自动脱敏。
- 场景 2：AI 生成 `rm -rf /data` 命令。无 skill 时在 skip-permissions 模式下直接执行；有 skill 后触发告警。
- 场景 3：多次交互后规则库逐步积累，展示规则从无到有的学习过程。

### 规则学习的简化方案

如果五周时间不够实现自动建议，可以简化为手动添加规则。核心价值（规则库驱动的脱敏和命令审查）不受影响，自动学习作为未来工作在报告中讨论即可。
